TLS Configuration

TLS terminates in two independent places in this stack — the static web frontend behind CloudFront, and the API behind a Caddy reverse proxy on EC2. They use different certificate authorities, renewal models, and minimum protocol floors. This document covers both, plus the encryption-in-transit and at-rest posture of supporting services.

Termination topology

            ┌──────────────── Web path ────────────────┐
browser ──TLS──▶ CloudFront ──HTTPS(AWS net)──▶ S3 (notism-web-prod, OAC)
  (redirect-to-https)   default *.cloudfront.net cert

            ┌──────────────── API path ────────────────┐
browser ──TLS──▶ Caddy :443 ──HTTP(docker net)──▶ api:5000 (Kestrel)
  notism-api.duckdns.org   Let's Encrypt cert        plaintext, internal only

In both paths TLS is terminated at the edge; the hop from edge to origin is either the trusted AWS backbone (CloudFront → S3) or the private Docker bridge network (Caddy → api), so the API container itself serves plain HTTP on :5000 and is never exposed directly.

API backend — Caddy on EC2

Defined in docker-compose.yml + Caddyfile, running on the EC2 instance (terraform/ec2.tf).

# Caddyfile
notism-api.duckdns.org {
    reverse_proxy notism-api:5000
}
  • Automatic HTTPS. Caddy provisions and renews a Let's Encrypt certificate for notism-api.duckdns.org via the ACME protocol on first boot — no manual certs, no certbot cron.
  • Modern defaults. Caddy negotiates TLS 1.2/1.3 only, with secure cipher suites and OCSP stapling out of the box. No explicit tuning needed.
  • Cert persistence. The caddy_data Docker volume stores issued certs and ACME account keys, so container restarts do not re-trigger issuance (and avoid Let's Encrypt rate limits).
  • Ports. Caddy binds 80 (ACME HTTP-01 challenge + HTTP→HTTPS redirect) and 443; the api service only exposes 5000 on the internal network — not published to the host.
  • Dynamic DNS. The hostname is a DuckDNS record pointing at the EC2 public IP.

nginx.conf — development only

Not part of the production TLS path. The repo's nginx.conf listens on plaintext :3000, proxies to api:5000, and exposes /swagger — no TLS at all. It is a local/dev reverse-proxy alternative. Production uses Caddy (docker-compose.yml), which is what provides HTTPS. Don't confuse the two.

Findings & hardening

SeverityFindingRecommendation
High CloudFront minimum_protocol_version = "TLSv1" permits deprecated TLS 1.0/1.1 for the web frontend. Attach a custom domain + ACM certificate and set minimum_protocol_version = "TLSv1.2_2021". The floor cannot be raised while using the default *.cloudfront.net cert.
Med SSH (22) open to 0.0.0.0/0. Restrict to a bastion/office CIDR, or replace with SSM Session Manager and drop the rule.
Low S3 buckets use SSE-S3 (AES256), not SSE-KMS. For audited key rotation / access logging on objects, switch sensitive buckets to aws:kms.
Good API TLS via Caddy — auto-issued, auto-renewed, TLS 1.2/1.3, internal API never exposed. No action. Keep caddy_data volume backed up to avoid re-issuance.
